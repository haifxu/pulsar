Architecture Overview

At the highest level, a Pulsar instance is composed of one or more Pulsar clusters. Clusters within an instance can replicate data amongst themselves.

一個 Pulsar 集群由下面三部分組成：

  • 一個或者多個 brokers 負責處理和負載均衡從 producer 源源不斷發送出的信息，並將他們發送給 consumer。它與信息存儲元交互來處理相應的任務，Pulsar 將信息資料存儲在 BookKeeper 實例中 (又稱之為 bookies)。它依賴 ZooKeeper 集群處理特定的任務。
  • 包含一個或者多個 Bookies 的 BookKeeper 集群，它主要負責信息的持久化存儲。
  • A ZooKeeper cluster specific to that cluster handles coordination tasks between Pulsar clusters.

Brokers

Pulsar 的 broker 是一個無狀態組件, 主要負責運行另外的兩個組件:

  • An HTTP server that exposes a REST API for both administrative tasks and topic lookup for producers and consumers. The producers connect to the brokers to publish messages and the consumers connect to the brokers to consume the messages.
  • 一個調度分發器, 它是異步的 TCP 服務器，通過自定義 二進制協議應用於所有相關的數據傳輸。

Messages are typically dispatched out of a managed ledger cache for the sake of performance, unless the backlog exceeds the cache size. 如果積壓的消息對於緩存來說太大了, 則 broker 將開始從BookKeeper 那裡讀取 Entries（Entry 同樣是 BookKeeper 中的概念，相當於一條記錄）。

最後，為了支持全局 Topic 信息異地複製，broker 會控制 Replicators 追踪本地發佈的條目，並把這些條目用Java client 重新發佈到其他區域

如何管理 Pulsar broker， 請參考 brokers 指南

Clusters

A Pulsar instance consists of one or more Pulsar clusters. Clusters, in turn, consist of:

  • 一個或者多個 Pulsar broker
  • 一個 ZooKeeper quorum，用於集群級別的配置和協調。
  • 一組 BookKeeper 的 bookie 用於信息的持久化存儲。

集群間可以通過異地複製進行消息同步。

如何管理 Pulsar 集群，請參考 clusters 指南。

元數據存儲

The Pulsar metadata store maintains all the metadata of a Pulsar cluster, such as topic metadata, schema, broker load data, and so on. Pulsar uses Apache ZooKeeper for metadata storage, cluster configuration, and coordination. The Pulsar metadata store can be deployed on a separate ZooKeeper cluster or deployed on an existing ZooKeeper cluster. You can use one ZooKeeper cluster for both Pulsar metadata store and BookKeeper metadata store. If you want to deploy Pulsar brokers connected to an existing BookKeeper cluster, you need to deploy separate ZooKeeper clusters for Pulsar metadata store and BookKeeper metadata store respectively.

In a Pulsar instance:

  • 配置與仲裁存儲: 存儲租戶，命名空間和其他需要全局一致的配置項。
  • Each cluster has its own local ZooKeeper ensemble that stores cluster-specific configuration and coordination such as which brokers are responsible for which topics as well as ownership metadata, broker load reports, BookKeeper ledger metadata, and more.

Configuration store

The configuration store maintains all the configurations of a Pulsar instance, such as clusters, tenants, namespaces, partitioned topic related configurations, and so on. A Pulsar instance can have a single local cluster, multiple local clusters, or multiple cross-region clusters. Consequently, the configuration store can share the configurations across multiple clusters under a Pulsar instance. The configuration store can be deployed on a separate ZooKeeper cluster or deployed on an existing ZooKeeper cluster.

持久化存儲

Pulsar provides guaranteed message delivery for applications. If a message successfully reaches a Pulsar broker, it will be delivered to its intended target.

為了提供這種保證，未確認送達的信息需要持久化存儲直到它們被確認送達。 This mode of messaging is commonly called persistent messaging. 在 Pulsar 內部，所有信息都被保存並同步N份，例如，2個服務器保存四份，每個服務器上面都有鏡像的RAID存儲。

Apache BookKeeper

Pulsar 用 Apache BookKeeper 作為持久化存儲。 BookKeeper 是一個分佈式的預寫日誌（WAL）系統，有如下幾個特性特別適合 Pulsar 的應用場景：

  • It enables Pulsar to utilize many independent logs, called ledgers. Multiple ledgers can be created for topics over time.
  • 為按條目複製的順序數據提供了非常高效的存儲。
  • 它保證了多系統掛掉時 ledgers 的讀取一致性。
  • 提供不同的 bookie 之間均勻的 IO 分佈的特性。
  • It's horizontally scalable in both capacity and throughput. Capacity can be immediately increased by adding more bookies to a cluster.
  • Bookie 被設計成可以承載數千的並發讀寫的ledger。 搭配使用多個硬碟設備，一個用於日誌，另一個用於一般存儲，這樣 bookie 可以將讀操作的影響和對於寫操作的延遲分隔開。

In addition to message data, cursors are also persistently stored in BookKeeper. Cursors 是消費端 subscription 消費的位置。 BookKeeper 讓 Pulsar 可以用一種可擴展的方式存儲消費位置。

At the moment, Pulsar supports persistent message storage. This accounts for the persistent in all topic names. Here's an example:

persistent://my-tenant/my-namespace/my-topic

Pulsar 也支持臨時消息（非持久消息）存儲。

Ledgers

Ledger 是一個 append-only 數據結構，並且只有一個寫入器，這個寫入器負責多個 BookKeeper 存儲節點（就是 Bookie）的寫入。 Ledger 的條目會被複製到多個 bookie。 Ledger 本身有著非常簡單的語義：

  • Pulsar broker 可以創建 ledeger，添加內容到 ledger 和關閉ledger。
  • 當一個 ledger 被關閉後，除非明確的要寫數據或者是因為寫入器掛掉導致 ledger 關閉，這個 ledger 只會以讀取模式打開。
  • 最後，當 ledger 中的條目不再有用的時候，整個 legder 可以被刪除（ledger 分佈是跨 bookie 的）。

Ledger 讀取一致性

BookKeeper 的主要優勢在於它能在有系統故障時保證讀取一致性。 由於 ledger 只能被一個進程寫入（之前提的寫入器進程），這樣這個進程在寫入時不會有衝突，從而寫入會非常高效。 在故障之後，ledger 會啟動一個恢復進程來確定 ledger 的最終狀態並確認最後提交到日誌的是哪一個條目。 在這之後，能保證所有的 ledger 讀進程讀取到相同的內容。

Managed ledgers

Given that Bookkeeper ledgers provide a single log abstraction, a library was developed on top of the ledger called the managed ledger that represents the storage layer for a single topic. 一個 managed ledger 就是信息流的抽象，有一個寫入器進程不斷在信息流結尾添加信息，並且有多個 cursors 消費這個信息流，每個 cursor 有自己的消費位置記錄。

Internally, a single managed ledger uses multiple BookKeeper ledgers to store the data. There are two reasons to have multiple ledgers:

  1. 在故障之後，原有的某個 ledger 不能再寫了，需要創建一個新的。
  2. A ledger can be deleted when all cursors have consumed the messages it contains. This allows for periodic rollover of ledgers.

日誌存儲

In BookKeeper, journal files contain BookKeeper transaction logs. 在更新到 ledger 之前，bookie 需要確保描述這個更新的事務被寫到持久（非揮發）存儲上面。 當 bookie 剛啟動和舊的日誌文件大小達到上限（由journalMaxSizeMB 參數配置）的時候，新的日誌文件資料檔會被創建起來。

Pulsar proxy

Pulsar client 和 Pulsar 集群交互的一種方式就是直接連接到 Pulsar broker 。 然而在某些情況下，這種直連既不可行也不可取，因為客戶端並不知道 broker 的地址。 例如在雲端環境或者像 Kubernetes 以及其它類似的系統上面運行 Pulsar，直連 broker 就基本上不可能了。

The Pulsar proxy provides a solution to this problem by acting as a single gateway for all of the brokers in a cluster. 如果選擇運行 Pulsar Proxy（這是可選的），所有的客戶端連接將會通過這個代理而不是直接與 brokers 通信。

為了性能和容錯，你可以運行任意多個 Pulsar proxy。

從架構上來看，Pulsar Proxy 從 ZooKeeper 上面讀取它所需要的所有訊息。 當在一個節點啟動代理時，只需要提供用於集群獨有和實例範圍的配置存儲的 ZooKeeper 連接字串。 Here's an example:

$ bin/pulsar proxy \
  --zookeeper-servers zk-0,zk-1,zk-2 \
  --configuration-store-servers zk-0,zk-1,zk-2

Pulsar proxy 文檔

如何使用 Pulsar proxy，參考 Pulsar proxy 管理指南。

關於 Pulsar proxy 有一些比較重要的注意點：

  • Connecting clients don't need to provide any specific configuration to use the Pulsar proxy. 除了更新用於服務URL的IP之外，你不需要為現有的應用更新客戶端配置（例如你在 Pulsar proxy 上層架設運行了負載均衡器）。
  • Pulsar proxy 支持 TLS 加密 和 認證。

Service discovery

連接到 Pulsar broker 的客戶端需要能夠使用單個 URL 與整個 Pulsar 實例進行通信。 Pulsar 內部提供了服務發現的機制，你可以通過配置Pulsar實例指南來進行設定。

你也可以用你自己的服務發現系統。 If you use your own system, there is just one requirement: when a client performs an HTTP request to an endpoint, such as http://pulsar.us-west.example.com:8080, the client needs to be redirected to some active broker in the desired cluster, whether via DNS, an HTTP or IP redirect, or some other means.

下面這張圖展示了 Pulsar 服務發現機制：

圖中，Pulsar 集群可以通過一個 DNS 名稱來解析位址：pulsar-cluster.acme.com。 例如 Python客戶端，可以像這樣訪問這個 Pulsar 集群：

from pulsar import Client

client = Client('pulsar://pulsar-cluster.acme.com:6650')

Note In Pulsar, each topic is handled by only one broker. Initial requests from a client to read, update or delete a topic are sent to a broker that may not be the topic owner. If the broker cannot handle the request for this topic, it redirects the request to the appropriate broker.
