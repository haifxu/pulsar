Architecture Overview

At the highest level, a Pulsar instance is composed of one or more Pulsar clusters. Clusters within an instance can replicate data amongst themselves.

Pulsar 클러스터의 내부:

  • 하나 이상의 브로커는 프로듀서로부터 들어오는 메시지들을 처리하고 로드밸런싱하여 소지자에게 전송 합니다. 다양항 조정 작업들을 처리하기 위해 구성 된 저장소와 상호 작용하고 BookKeeper 인스턴스 (일명 Bookies) 에 메시지를 저장합니다. 그리고 클러스터에 특화 된 특정 작업 등을 위해 ZooKeeper 클러스터에 의존합니다.
  • 하나 이상의 bookies로 구성된 BookKeeper 클러스터는 메시지의 영구 저장을 처리합니다.
  • A ZooKeeper cluster specific to that cluster handles coordination tasks between Pulsar clusters.

아래 다이어그램은 Pulsar 클러스터를 보여줍니다:

보다 광범위한 인스턴스 수준에서, 인스턴스 전체에 ZooKeeper 클러스터는 geo-replication와 같은 여러 클러스터와 관련된 조정 작업을 처리합니다.

브로커

Pulsar 메시지 브로커는 다른 두 가지 주요 구성 요소를 실행하는 책임이 있는 stateless 컴포넌트 입니다:

  • An HTTP server that exposes a REST API for both administrative tasks and topic lookup for producers and consumers. The producers connect to the brokers to publish messages and the consumers connect to the brokers to consume the messages.
  • 디스패처: 비동기 TCP 서버인 디스패처는 사용자 정의 이진 프로토콜을 통해 모든 데이터 전송에 사용됩니다.

Messages are typically dispatched out of a managed ledger cache for the sake of performance, unless the backlog exceeds the cache size. Backlog가 캐시에 비해 너무 커지면 브로커는 BookKeeper에서 엔트리(레코드) 를 읽기 시작합니다.

마지막으로, 글로벌 topic에 대한 geo-replication을 지원하기 위해 브로커는 로컬 영역에 공개 된 엔트리(레코드)를 테일링하고 Pulsar Java client library를 사용하여 리모드 리전 클러스터에 리-퍼블리싱하는 리플리케이터를 관리합니다.

Pulsar 브로커 관리에 대한 가이드는 brokers 가이드를 참조하십시오.

클러스터

A Pulsar instance consists of one or more Pulsar clusters. Clusters, in turn, consist of:

  • 하나 이상의 Pulsar 브로커
  • 클러스터 수준 구성 및 조정에 사용되는 ZooKeeper 쿼럼
  • 메시지의 영구 저장소로 사용되는 bookies 앙상블(Bookeeper)

클러스터는 geo-replication을 사용하여 스스로 메시지를 복제 할 수 있습니다.

Pulsar 클러스터 관리에 대한 가이드는 클러스터 가이드를 참조하십시오.

Metadata store

The Pulsar metadata store maintains all the metadata of a Pulsar cluster, such as topic metadata, schema, broker load data, and so on. Pulsar uses Apache ZooKeeper for metadata storage, cluster configuration, and coordination. The Pulsar metadata store can be deployed on a separate ZooKeeper cluster or deployed on an existing ZooKeeper cluster. You can use one ZooKeeper cluster for both Pulsar metadata store and BookKeeper metadata store. If you want to deploy Pulsar brokers connected to an existing BookKeeper cluster, you need to deploy separate ZooKeeper clusters for Pulsar metadata store and BookKeeper metadata store respectively.

In a Pulsar instance:

  • 구성(configuration) 저장소인 쿼럼(Zookeeper) 은 테넌트, 네임 스페이스 및 전체적으로 일관성이 필요한 다른 엔터티에 대한 구성을 저장합니다.
  • Each cluster has its own local ZooKeeper ensemble that stores cluster-specific configuration and coordination such as which brokers are responsible for which topics as well as ownership metadata, broker load reports, BookKeeper ledger metadata, and more.

Configuration store

The configuration store maintains all the configurations of a Pulsar instance, such as clusters, tenants, namespaces, partitioned topic related configurations, and so on. A Pulsar instance can have a single local cluster, multiple local clusters, or multiple cross-region clusters. Consequently, the configuration store can share the configurations across multiple clusters under a Pulsar instance. The configuration store can be deployed on a separate ZooKeeper cluster or deployed on an existing ZooKeeper cluster.

Persistent storage

Pulsar provides guaranteed message delivery for applications. If a message successfully reaches a Pulsar broker, it will be delivered to its intended target.

이 전송 보증을 제공하기 위하여는 컨슈머에게 승인 된 메시지가 전달되기 전까지 승인되지 않은 메시지가 내구성있는 방식으로 저장되어야합니다. This mode of messaging is commonly called persistent messaging. Pulsar에서는 모든 메시지의 N 개 사본이 디스크에 저장 및 동기화됩니다. 예: 각 서버에 미러링 된 RAID 볼륨이있는 두 서버에서 4 개의 사본.

Apache BookKeeper

Pulsar는 영구 메시지 저장을 위해 Apache BookKeeper라는 시스템을 사용합니다. BookKeeper는 Pulsar에 여러 가지 중요한 이점을 제공하는 분산 write-ahead log (WAL) 시스템입니다.

  • It enables Pulsar to utilize many independent logs, called ledgers. Multiple ledgers can be created for topics over time.
  • 이것은 엔트리(레코드) 복제를 처리하는 순차적 데이터를 위한 매우 효율적인 저장소를 제공합니다.
  • 다양한 시스템 오류가있는 경우 원장(ledgers) 의 읽기 일관성을 보장합니다.
  • Bookies에 걸쳐 I/O의 균일 한 분배를 제공합니다.
  • It's horizontally scalable in both capacity and throughput. Capacity can be immediately increased by adding more bookies to a cluster.
  • Bookies는 동시에 읽고 쓰는 수천 개의 원장(ledgers) 을 처리하도록 설계 되었습니다. 여러 디스크 장치--저널용 하나와 일반 저장소용 하나--를 사용함으로써, Bookies는 읽기 작업의 영향을 지속적인 쓰기 작업의 대기 시간과 분리 할 수 ​​있습니다.

In addition to message data, cursors are also persistently stored in BookKeeper. 커서는 소비자의 구독 위치입니다. Pulsar는 BookKeeper를 사용하여 확장 가능한 방식으로 consumers 위치를 저장 할 수 있습니다. Pulsar는 BookKeeper를 사용하여 확장 가능한 방식으로 컨슈머 위치를 저장 할 수 있습니다.

At the moment, Pulsar supports persistent message storage. This accounts for the persistent in all topic names. Here's an example:

persistent://my-tenant/my-namespace/my-topic

Pulsar는 또한 임시 (비 지속적) 메시지 저장을 지원합니다.

아래 다이어그램에서 브로커와 bookies가 어떻게 상호 작용하는지에 대한 그림을 볼 수 있습니다.

Ledgers

원장(ledger) 은 여러 BookKeeper 스토리지 노드 또는 bookies에 할당 된 단일 작성기가있는 추가 전용 데이터 구조입니다. 원장 엔트리(레코드) 은 여러 bookies로 복제됩니다. 원장 자체에는 매우 간단한 의미구조가 있습니다:

  • Pulsar 브로커는 원장을 생성하고 원장에 엔트리를 추가 한 후 원장을 닫을 수 있습니다.
  • 원장이 닫히면 --명시 적으로 또는 기록기 프로세스가 충돌하여-- 읽기 전용 모드로만 열 수 있습니다.
  • 마지막으로, 원장의 엔트리가 더 이상 필요하지 않으면 전체 원장을 시스템에서 삭제 할 수 있습니다 (모든 bookies에서).

Ledger read consistency

Bookkeeper의 주요 장점은 장애가있는 경우 원장의 읽기 일관성을 보장한다는 것 입니다. Ledger은 단일 프로세스로만 작성 되므로 해당 프로세스는 합의를 얻지 않고도 엔트리를 매우 효율적으로 추가 할 수 있습니다. 장애가 발생하면 ledger은 복구 프로세스를 거쳐 ledger의 상태를 마무리하고 로그에 마지막으로 커밋 된 엔트리를 설정합니다. 그후, ledger의 모든 리더(reader) 는 동일한 내용을 볼 수 있습니다.

Managed ledgers

Given that Bookkeeper ledgers provide a single log abstraction, a library was developed on top of the ledger called the managed ledger that represents the storage layer for a single topic. Managed ledger은 메시지 스트림의 추상화로, 스트림의 끝에 엔트리를 지속적으로 추가하는 단일 기록기 프로세스가 있으며 이 스트림을 소비하는 여러 커서가 있으며 각 커서에는 고유한 소비 위치가 있습니다.

Internally, a single managed ledger uses multiple BookKeeper ledgers to store the data. There are two reasons to have multiple ledgers:

  1. 실패 후, 원래 ledger는 더 이상 쓸 수 없으며 새 ledger을 만들어야 합니다.
  2. A ledger can be deleted when all cursors have consumed the messages it contains. This allows for periodic rollover of ledgers.

Journal storage

In BookKeeper, journal files contain BookKeeper transaction logs. Ledger를 업데이트하기 전에, bookie는 업데이트를 설명하는 트랜잭션이 영구 (비 휘발성) 스토리지에 기록 되도록 해야합니다. Bookie가 시작 되거나 이전 저널 파일이 저널 파일 크기 임계 값 (journalMaxSizeMB 매개 변수를 사용하여 구성 됨)에 도달하면 새 저널 파일이 작성 됩니다.

Pulsar proxy

Pulsar 클라이언트가 Pulsar 클러스터 와 상호 작용하는 한 가지 방법은 Pulsar 메시지 브로커에 직접 연결하는 것입니다. 그러나 클라이언트가 브로커 주소에 직접 액세스 할 수 없기 때문에 이러한 종류의 직접 연결은 불가능하거나 바람직하지 않은 경우도 있습니다. 예를 들어 클라우드 환경이나 Kubernetes 또는 유사한 플랫폼에서 Pulsar를 실행하는 경우 브로커에 직접 클라이언트를 연결할 수 없습니다.

The Pulsar proxy provides a solution to this problem by acting as a single gateway for all of the brokers in a cluster. Pulsar proxy를 실행하는 경우 (선택 사항 임) Pulsar 클러스터와의 모든 클라이언트 연결은 브로커와 통신하지 않고 프록시를 통과합니다.

성능 및 내결함성을 위해 원하는 만큼 Pulsar proxy 인스턴스를 실행 할 수 있습니다.

구조적으로 Pulsar proxy는 ZooKeeper에서 필요한 모든 정보를 얻습니다. 시스템에서 프록시를 시작할 때, 각 인스턴스 클러스터와 전체 구성 저장소 클러스터에 ZooKeeper 연결 문자열만 제공하면 됩니다. Here's an example:

$ bin/pulsar proxy \
  --zookeeper-servers zk-0,zk-1,zk-2 \
  --configuration-store-servers zk-0,zk-1,zk-2

Pulsar proxy docs

Pulsar 프록시 사용에 대한 설명서는 Pulsar 프록시 관리자 설명서를 참조하십시오.

Pulsar proxy에 대해 알아야 할 중요한 사항:

  • Connecting clients don't need to provide any specific configuration to use the Pulsar proxy. 서비스 URL에 사용 된 IP를 업데이트하는 것 외에 (예: Pulsar 프록시를 통해로드 밸런서를 실행하는 경우) 기존 애플리케이션의 클라이언트 구성을 업데이트 할 필요가 없습니다.
  • Pulsar proxy에서 TLS 암호화 및 인증지원

Service discovery

Pulsar 브로커에 연결하는 클라이언트는 단일 URL을 사용하여 전체 Pulsar 인스턴스와 통신 할 수 있어야합니다. Pulsar는 Pulsar 인스턴스 배포 가이드의 지침을 사용하여 설정할 수있는 기본 제공 서비스 검색 메커니즘을 제공합니다.

원하는 경우 고유 한 서비스 검색 시스템을 사용할 수 있습니다. If you use your own system, there is just one requirement: when a client performs an HTTP request to an endpoint, such as http://pulsar.us-west.example.com:8080, the client needs to be redirected to some active broker in the desired cluster, whether via DNS, an HTTP or IP redirect, or some other means.

아래 다이어그램은 Pulsar 서비스 검색을 보여줍니다:

이 다이어그램에서 Pulsar 클러스터는 단일 DNS 이름인 pulsar-cluster.acme.com을 통해 주소를 지정할 수 있습니다. 예를 들어 Python 클라이언트는 다음과 같이이 Pulsar 클러스터에 액세스 할 수 있습니다:

from pulsar import Client

client = Client('pulsar://pulsar-cluster.acme.com:6650')

Note In Pulsar, each topic is handled by only one broker. Initial requests from a client to read, update or delete a topic are sent to a broker that may not be the topic owner. If the broker cannot handle the request for this topic, it redirects the request to the appropriate broker.
